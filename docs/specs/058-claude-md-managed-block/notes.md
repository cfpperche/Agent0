## Design decisions

_Choices made where the spec/plan was ambiguous. The decision itself + why this option over others considered in the moment._

### 2026-05-19 — parent — region-replacement vs body-divergence — heading-set diff is NOT divergence

spec 058 scenarios 1 (paired markers → replace) and 9 (region-divergence → refuse) describe paired markers + `--apply` but expect different outcomes. The mechanical primitive can't distinguish "Agent0 added a section" from "fork edited a section body" without state. Resolution: the divergence guard fires on **per-section body divergence** of the heading INTERSECTION (`_check_section_divergence` against the region), NOT on overall region content equality. Heading-only diffs (fork has X, Agent0 doesn't, or vice versa) are the intended target of wholesale replace — including orphan removal. This makes scenarios 1 and 9 cleanly distinguishable: scenario 1 = src has new heading C that fork doesn't have → intersection {A,B} bodies match → replace; scenario 9 = fork edited body of shared heading A → intersection {A} body differs → refuse.

### 2026-05-19 — parent — fork's project sections classified by Agent0-region heading-set, not full-Agent0 heading-set

Discovered while debugging test 06 regression: Agent0's full CLAUDE.md `## ` set includes BOTH project-narrative sections (Overview, Stack, Build & test, Conventions, Gotchas — the *template* sections that forks customize) AND capacity sections (Spec-driven development, etc.). A naive `_check_section_divergence(src, dst)` would flag fork's `## Overview` body ("my fork overview") as diverged from Agent0's placeholder ("_Brief description..._"). Resolution: divergence check compares fork sections against Agent0's REGION (between markers) only. Outside the region = project-narrative namespace, fork-owned, no divergence guard. This is also why `_generate_migration_candidate` no-ops when Agent0 source has no markers — without markers, the namespace boundary is undefined.

### 2026-05-19 — parent — candidate comment-block is a UX leak after ratification

The candidate file embeds a leading HTML comment ("Generated by sync-harness.sh on YYYY-MM-DD... mv to ratify") for the operator's review context. After `mv candidate CLAUDE.md`, this comment becomes a permanent stale header in CLAUDE.md. acmeyard dogfood demonstrated this — operator (me) had to manually delete the comment post-ratification. The spec's open Q4 anticipated this ("Should migration candidate include a leading comment block explaining itself? Lean yes for v1") but didn't resolve the post-ratification cleanup. Acceptable for v1 since the comment is harmless markdown noise and one-time-cost per fork migration; revisit if multiple forks complain. Could be addressed in v2 by either: (a) emitting the explanation to stderr only (operator reads the advisory line, not a file header) or (b) detecting-and-stripping the leading sync-harness comment on first paired-state sync.

## Deviations

_Places where implementation intentionally departed from `plan.md`. The departure + the reason it was necessary or better._

### 2026-05-19 — parent — no `_helpers.sh` test library

Plan task 22 specified `.claude/tests/harness-sync/_helpers.sh` exporting `make_wrapped_claude_md()` and `make_legacy_claude_md()` helpers to deduplicate test fixtures. Skipped during implementation: heredocs in each test file are clearer than a function call for one-shot fixtures, and the boilerplate cost (~10 lines per test) was less than the maintenance cost of a shared library that few callers exercise. Each of tests 16-22 inlines its setup. Revisit if test 23+ shows real repetition.

### 2026-05-19 — parent — task 12 destructive-rename dogfood not executed

Plan day 1 task 12 specified renaming an Agent0 capacity title temporarily, sync-ing, verifying, then reverting. Skipped: tests 16 and 22 already exercise the same code path (paired markers + heading-set diff → region replace), and the destructive-rename on live Agent0 source carried real risk of leaving the source in inconsistent state if interrupted. The synthetic-fixture coverage in `.claude/tests/harness-sync/16-*.sh` and `22-*.sh` is mechanically equivalent and reversible.

## Tradeoffs

_Alternatives weighed during implementation (not at plan time). The chosen path + what was given up + why the tradeoff was worth it._

### 2026-05-19 — parent — bash `grep -c` count-via-fallback pattern

Encountered three printf-style bash gotchas during implementation, each adding one line of friction:
1. `grep -c ... || echo 0` outputs `"0\n0"` when no match (grep emits its own "0" then the OR fires). Fix: `grep -c ... || true` + `[ -z "$x" ] && x=0` (preserves grep's "0", salvages exit).
2. `printf '-->\n'` fails with "invalid option" (printf treats `-` as flag indicator). Fix: `printf '%s\n' '-->'` for literal lines starting with `-`.
3. `printf '- \`%s\`\n' "$title"` same issue. Fix: `printf -- '- \`%s\`\n' "$title"` (explicit end-of-options marker).

The chosen idioms are the bash-3.2-portable canonical workarounds. Trading ergonomic format strings for robustness against tokens that look like flags. Documented in plan.md Risk #3.
