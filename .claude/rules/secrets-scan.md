# Secrets scan

A two-layer capacity that keeps credentials out of the repo, plus a third soft-advisory hook on edits. The discipline is anchored on the same primitives as the governance and delegation gates — stdin JSON payload, `# OVERRIDE:` escape, JSONL audit log — so a reader who already knows those two gates can read this one in 30 seconds. Specs: `docs/specs/006-secrets-scan/` (original capacity) and `docs/specs/007-secrets-scan-timing/` (layered timing fix).

## What fires, what advises

**Primary block — `.githooks/pre-commit` (native git hook).** Activated per-fork by running `git config core.hooksPath .githooks` once after `git init`. Fires inside git's own commit process *after* staging is finalised, so the scan always sees the real index — including the case where the agent ran `git add ... && git commit ...` as one Bash invocation (the timing bug spec 007 closed). The script runs `gitleaks git --pre-commit --staged --no-banner --report-format=json --report-path=<mktemp>` from the repo root, parses the report's array length defensively, and decides: 0 findings → audit `decision: "allow"` + exit 0; ≥1 findings + `CLAUDE_SECRETS_OVERRIDE_REASON` env var set → audit `decision: "override"` with the reason populated + exit 0; ≥1 findings + no env var → audit `decision: "block"` + exit 1 (native pre-commit convention, NOT exit 2). Empty staged diff (e.g. `git commit --allow-empty`) → audit `decision: "allow-empty"` distinguishing it from a real clean scan. Missing gitleaks → audit `decision: "skip-no-engine"` + fail-open. Parse failure on the gitleaks report → audit `decision: "allow-parse-error"` so forensic analysis can tell "scan ran clean" apart from "scan output unparseable". `scan_mode: "native-pre-commit"`, `session_id: null`, `agent_id: null` (native hook has no Claude Code payload).

**Preflight shape-gate — `PreToolUse(Bash)` → `.claude/hooks/secrets-scan.sh`.** Registered with a broad matcher that fires on any Bash command containing `git`; the script short-circuits unless the command is a real `git commit` invocation (covers `git commit`, `git  commit` with double space, `git -C <path> commit`, `git commit --amend`). On a non-commit shape, audits `decision: "skip-not-commit"` + exit 0. On a real `git commit`, it does NOT call gitleaks — that responsibility is now exclusively the native layer's. Instead it (a) parses the `# OVERRIDE: <reason ≥10 chars>` marker (start-of-line anchored), (b) rejects dangerous command shapes when no valid override is present: compound `&& git commit`, compound `; git commit`, `git commit -a` / `-am` / `-ma`, `git commit --no-verify`. Each rejection emits a verbatim stderr template with the corrected separated form (the Claude Code issue #24327 mitigation — see § *Gotchas*) and audits `decision: "reject-shape"` with a `cmd_shape` detail field. (c) When a valid override is present, the preflight emits the override pass-through: rewrites the command to prepend a `export CLAUDE_SECRETS_OVERRIDE_REASON='<reason>'; ` statement and outputs JSON `{"hookSpecificOutput":{"hookEventName":"PreToolUse","updatedInput":{"command":"<rewritten>"}}}` on stdout for the harness to execute. The native hook then inherits the env var and audits `decision: "override"`. (d) When the shape is clean and no override marker is present, audits `decision: "passthrough"` and exits 0 silently — the harness runs the command unchanged, and the native hook handles findings. Every preflight audit row carries `scan_mode: "preflight"` and includes `session_id` + `agent_id` (which the native layer cannot capture).

**Soft advisory — `PostToolUse(Edit|Write|MultiEdit)` → `.claude/hooks/secrets-advise.sh`.** Opt-in via `CLAUDE_SECRETS_ADVISE_ON_EDIT=1`; unset means the hook exits 0 silently. Even when enabled, the advisory is *sub-agent only* — the script reads `agent_id` from the stdin payload and exits 0 when absent (parent edits), mirroring the actor split used by the post-edit validator (see `.claude/rules/delegation.md` § *Post-edit validator loop*). On a delegated edit, gitleaks runs against the new content in a temp dir; each finding becomes one `secrets-advisory: <detector> at <file>:<line>` line on stderr. The hook always exits 0 — advisories never block, never revert the edit, and never enter the commit audit log. Same shape as the TDD `tdd-advisory:` prefix documented in `.claude/rules/tdd.md` § *Reading the validator advisory*.

The matcher-is-broad / script-is-precise split on the preflight is deliberate — same rationale as spec 006. False fires short-circuit cheap (no audit, one bash startup); missed fires would be unscanned commits.

## Override grammar

A line in the `tool_input.command` string matching `^[[:space:]]*# OVERRIDE: <reason>` skips the shape-rejection. **Start-of-line anchored** (with optional leading whitespace), copied verbatim from the 002-delegation fix that closed a false-positive where `# OVERRIDE:` appearing inside a quoted string was matching. Inline-trailing markers on a single line are NOT accepted, even when "intended" — they re-open the regression and were dropped during spec 007. The legitimate single-shape usage is a **two-line Bash command** with the marker on its own line:

```bash
git commit -m "land auth-test fixture"
# OVERRIDE: AWS key is a documented test vector for the auth fixture suite
```

Bash treats line 2 as a no-op comment when the command actually runs; the preflight sees line 2 as start-of-line text and matches. For compound shapes:

```bash
git add tests/fixtures/aws-key.txt && git commit -m "fixture"
# OVERRIDE: documentation test vector for AWS detector regression suite
```

Reason length is measured after trim. **`<reason>` must be at least 10 characters** — anything shorter (`# OVERRIDE: skip`, `# OVERRIDE: n/a`) is rejected with `secrets-scan: override reason must be ≥10 characters, got "<reason>"` on stderr and the command still blocks. The 10-char floor is the same as the governance and delegation gates.

### Env-var bridge

The override marker lives in `tool_input.command` — accessible to the preflight (which reads stdin JSON from the Claude Code harness) but invisible to the native git hook (which runs inside git's own process, has no Claude Code payload, and the shell strips `#`-comments before git ever runs). The bridge is `CLAUDE_SECRETS_OVERRIDE_REASON`: when the preflight parses a valid marker, it rewrites the command to prepend `export CLAUDE_SECRETS_OVERRIDE_REASON='<escaped-reason>'; ` (single-quoted; the reason is sanitised against shell injection via the close-escape-open idiom `'\''`). The rewritten command then runs in the harness's bash subprocess; the `export` statement makes the var inheritable; every chained command — including `git commit` — sees the var; the native hook reads it and audits `decision: "override"`.

The injection MUST be a standalone statement followed by `;`, NOT the inline prefix form `VAR=val cmd`. The prefix form scopes the assignment to the single command it prefixes, so on `VAR=val git add foo && git commit -m "..."` the env var reaches `git add` but NOT the chained `git commit` — the native hook on commit sees no override and blocks. This was the spec-007 latent bug fixed in commit `4b47a42`; the V4 test (`tests/secrets-scan/04-override-allows.sh`) asserts the rewriting starts with `export CLAUDE_SECRETS_OVERRIDE_REASON=` as a regression guard.

Override semantics are surgical: the marker skips ONLY the preflight shape rejection (for compound / `-a` / `--no-verify` paths) AND the native block (when findings are present). The scan still runs, both audit rows are still written, and `override_reason` is populated. There is no silent bypass.

## Allowlist mechanics

Two complementary suppression mechanisms, both honored by gitleaks at scan time:

**`.gitleaks.toml` at repo root.** The shipped starter uses `[extend].useDefault = true` to inherit gitleaks' built-in detector set, then a `[[allowlists]]` block exposes the three suppression dimensions:

- `paths = ["tests/fixtures/**/*", "examples/secrets-demo.md"]` — glob patterns. Anything matching is skipped wholesale.
- `regexes = ['''AKIA[0-9A-Z]{16}''', '''ghp_[A-Za-z0-9]{36}''']` — TOML triple-quoted regex strings; matches in any file are suppressed.
- `commits = ["abc123def456..."]` — full SHAs. Useful when a historical commit is known-clean and the scan keeps re-flagging it via amend or rebase.

Add as many `[[allowlists]]` blocks as needed; each one is a logical OR. Schema reference: <https://github.com/gitleaks/gitleaks/blob/master/config/allowlist.go>. The starter ships with `paths = []` as a placeholder — replace with real exemptions or delete the block. Do **not** ship a `.secrets.baseline` (the detect-secrets pattern); that is a non-goal because Agent0 is a new-repo template with no legacy to freeze.

**Inline `gitleaks:allow`.** A comment containing `gitleaks:allow` on the same line as a high-entropy string suppresses that single finding without modifying `.gitleaks.toml`. The comment form is language-appropriate — `# gitleaks:allow` in shell/Python/TOML, `// gitleaks:allow` in JS/TS/Rust, `<!-- gitleaks:allow -->` in Markdown/HTML. Prefer inline `gitleaks:allow` for one-off lines (a fixture string in a test file); prefer `.gitleaks.toml` paths for whole directories of fixtures.

## Escape hatch

`CLAUDE_SKIP_SECRETS_SCAN=1` in the environment makes both hooks (preflight + native) exit 0 silently — no scan, no audit entry. Use it for throwaway scratch repos or local Q&A sessions where no commit is intended to land in shared history. Setting the variable in a long-lived shell config is the wrong escape — that disables the discipline permanently and silently. The override marker is the right tool for "this one commit needs to land despite the finding"; the env var is the right tool for "this whole session is throwaway".

The advisory side has its own env var with the inverse polarity: `CLAUDE_SECRETS_ADVISE_ON_EDIT=1` turns the on-edit advisory **on**. Default is off so the advisory does not surprise users who haven't read this doc.

When gitleaks itself is missing from `PATH`, both hooks fail open: one stderr line `secrets-scan: gitleaks not found, scan skipped` (native side audits `skip-no-engine`), the preflight's shape-gate logic still runs since it never depends on gitleaks. The template is stack-agnostic and the fork chooses whether to install the binary; a broken or absent engine must never permanently lock the agent out of committing.

## Audit log

`.claude/secrets-audit.jsonl`, gitignored, append-only, `flock`-atomic. Every invocation of either commit-time layer writes exactly one line. Decision values split cleanly by layer:

| Decision | Layer | Meaning |
| --- | --- | --- |
| `skip-not-commit` | preflight | Command does not match `git commit` shape; harness passes through |
| `passthrough` | preflight | Real `git commit`, clean shape, no override — fall through to native |
| `reject-shape` | preflight | Compound / `-a` / `--no-verify` without override; exits 2 with corrected template on stderr. `cmd_shape` field names which pattern matched (`compound-and`, `compound-semicolon`, `git-commit-dash-a`, `git-commit-no-verify`, `override-too-short`) |
| `override-pass-through` | preflight | Valid override marker parsed; command rewritten with `export ...;` prefix; `override_reason` populated |
| `allow` | native | gitleaks ran clean over a non-empty staged diff |
| `allow-empty` | native | `staged_files_count == 0` (e.g. `--allow-empty` commits); distinct from `allow` so the spec-006 silent-failure mode is unreproducible |
| `allow-parse-error` | native | gitleaks JSON unparseable; fail-open with forensic signal preserved (NOT collapsed into `allow`) |
| `override` | native | Findings present and `CLAUDE_SECRETS_OVERRIDE_REASON` env var set; reason populated |
| `block` | native | Findings present, no override, exit 1 (native pre-commit convention) |
| `skip-no-engine` | native | gitleaks missing from PATH; fail-open |

Both layers tag rows with `scan_mode` (`"preflight"` or `"native-pre-commit"`) so a single `jq` query can split or join across them. The preflight rows include `session_id` and `agent_id`; the native rows have those fields set to `null` because git's commit process is outside Claude Code's payload reach. The advisory hook does **not** write here — advisories are a stderr stream, not an audit stream. Read with `jq -c .` or `tail -f`.

## Gotchas

- **`core.hooksPath` activation is MANUAL by design — Lazarus 2025 vector.** Do NOT install `core.hooksPath .githooks` via a post-clone, post-checkout, or `git init` template hook. That automation pattern is the exact mechanism used by the 2025 Lazarus Group "Contagious Interview" campaign to deliver malware via poisoned developer repositories. The README's per-fork checklist is the right place for the line; the fork developer types it once after `git init`, consciously, with eyes open. The cost is one extra documented step; the benefit is that a malicious downstream fork cannot silently activate hooks just by being cloned.
- **`--no-verify` discipline.** Git's `--no-verify` flag bypasses the native `.githooks/pre-commit` entirely. The preflight rejects this shape with a verbatim stderr template so the agent can pattern-match the correction; the alternative is "user has to know to drop the flag" which is exactly the implicit behavior the layered design is meant to prevent. Removing the flag is the canonical fix; the override marker is the emergency exit when the bypass is deliberate.
- **Global `core.hooksPath` shadowing.** If the user has set `git config --global core.hooksPath ~/some/path` outside this repo, the repo-local `git config core.hooksPath .githooks` is what wins (repo config overrides global). However, a fork that *forgets* to run the install step inherits the global setting, which usually doesn't point at `.githooks` in this repo, and the native hook silently does not fire. The preflight's `passthrough` audit still records the commit attempt, so post-hoc forensics can detect this — but the live signal is missing. Verify with `git config --get core.hooksPath` returning `.githooks` (repo-scoped value) as part of the install step.
- **`git commit -a` shape rejection.** The `-a` flag auto-stages tracked-file modifications, which skips the explicit `git add` step. The preflight rejects `-a` (and bundles like `-am`, `-ma`) because the auto-stage happens inside git's pre-commit process — by then the audit trail is muddier and the override-marker bridge is harder to reason about. The corrected form (`git add -u` + `git commit -m "..."`) is in the verbatim stderr template. `--all` (long form) is NOT in the rejection list — only the short `-a` and its bundles.
- **Claude Code issue #24327: stderr ingestion on exit-2 blocks.** When a hook exits 2 with a stderr message, Claude Code's harness ingests the stderr into the agent's next-turn context as a "what went wrong" signal. Agents can pattern-match the stderr text to decide the next action. The shape-rejection templates therefore end with the EXACT corrected form (`git add <files>` newline `git commit -m "..."`) so the agent can copy-paste the correction without semantic reasoning. If the templates ever drift in wording, the agent's correction loop degrades from "mechanical pattern match" to "infer the intent from prose" — slower and more error-prone. Treat the stderr templates as a contract, not as friendly UI text.
- **Preflight false-positive on commit messages mentioning compound syntax.** The preflight's shape detector is regex-based and does NOT parse quoting. A commit message body containing the literal substring `&& git commit` — for example, a heredoc'd message documenting a bug fix that involves the compound shape — will be matched as compound-and and rejected. Mitigations: (a) reword the message body to avoid the literal substring (most common), (b) use the multi-line override marker form (legitimate when the literal is intentional). The same caveat applies to commit messages containing `# OVERRIDE:` in prose; the start-of-line anchor protects against most such cases but a marker on its own line within a heredoc would still match.
- **`# OVERRIDE: ...` survives the JSON payload.** Shell strips `# ...` comments before invoking the underlying binary, but both hooks receive the raw command string — the preflight reads from `tool_input.command` (stdin JSON), the native hook reads from the env var the preflight injected. Neither sees a stripped command. This is the same upstream-of-shell mechanic the governance and delegation gates rely on; verify experimentally if the payload shape changes in a future Claude Code release.
- **Parse gitleaks JSON defensively.** v8.x has been stable but field renames have happened in the past. `jq` with `// empty` fallbacks helps, but mind the `false`-vs-missing collapse documented in `.claude/rules/delegation.md` § *Gotchas (for hook maintainers)*. When parsing the report's array length, use the `if type == "array" then length else -1 end` shape and surface the raw gitleaks output to stderr on parse failure — that's the `allow-parse-error` audit value, separate from real `allow`, so forensics stay accurate.
- **FP noise on test fixtures.** Sub-agents writing realistic auth/payments fixtures will trip the detectors. The mitigations are layered: inline `gitleaks:allow` for one-off lines, `.gitleaks.toml` paths for whole fixture dirs, the override marker for a single commit, the env var for throwaway sessions, source-level string-splitting (e.g. `"AKIA""1234567890ABCDEF"` — adjacent strings concatenate at parse time, regex scanners see a non-matching source) when even inline allowlist comments would trip a third-party scanner. Pick the narrowest tool that fits — broad escapes accumulate as silent debt.
- **gitleaks version skew across forks.** Detector set and flag names evolve. The rule documents v8.20+ as the current floor (the `gitleaks git --pre-commit --staged` invocation form requires it; the deprecated `gitleaks protect --staged` shape from spec 006 is gone). The hook does not gate on a version check; `gitleaks --version` parse can fail silently and the fail-open path still works. If a fork wants a stricter floor, that is a project-local decision documented in their own CLAUDE.md, not a template concern.
- **Audit-log append atomicity uses `flock`.** Both layers can fire concurrently — a preflight on a Bash invocation running in parallel with a native hook on a different terminal's commit, for example. JSONL one-line appends are generally safe on Linux for writes ≤ `PIPE_BUF` (typically 4096 bytes), but `flock` is the explicit guarantee against partial-line interleaving. Both hooks copy the delegation-gate's pattern (and its § *Gotchas (for hook maintainers)* documents the **sticky `exec 9>file 2>/dev/null` redirect** trap that bit this codebase before).
- **The native layer is hot on first fork — IF the install step is run.** Unlike the TDD validator (inert until a stack is detected), the secrets-scan capacity activates as soon as a fork runs `git config core.hooksPath .githooks`. A first-time user surprised by a block on a fixture should read this doc top-to-bottom, not disable `core.hooksPath` or unset it permanently. The escape hatch (`CLAUDE_SKIP_SECRETS_SCAN=1`) and override marker are designed for exactly this case.
