# 143 — od-vendor-skills-remap — plan

_Drafted from `spec.md` on 2026-06-02. Update this file if implementation reveals the plan is wrong; do NOT silently diverge._

## Approach

A near-surgical manifest change plus a re-apply. The c128ffd5 advance left the skills vendored-path mapping pointing at the wrong upstream tree; this spec re-points it and lets the (now content-true, spec 141) `--apply` reconcile.

**The one substantive change:** in `vendor/open-design/MANIFEST.json`, the skill-bundle vendored-path's `src` becomes `"design-templates/"` (was `"skills/"`); `dst`, `kind`, `recursive` are unchanged. Then `bun scripts/sync-open-design.ts --apply` stages the upstream `design-templates/` tree into the unchanged dst `vendor/open-design/skills/`, recomputing that path's tree checksum. Because the tarball at `c128ffd5` is already cached under `runtime/od-sync/`, the apply is local (no re-download). The 31 pipeline bundles (`web-prototype`, `saas-landing`, +29) land at `vendor/open-design/skills/<bundle>/` exactly where the pipeline templates already read them — so **zero pipeline-template edits**. The pre-existing upstream creative `skills/` bundles (the 154-set the c128 advance wrote) become orphans; pruning them is spec 142's job, so `--verify` stays red on `skills/` after this spec (documented, expected).

**OQ resolutions (all settled here):**
- **OQ1 — vendor whole tree vs only the 31.** → **whole `design-templates/` tree** (729 files / 111 bundles). Consistent with spec 027's founder-ratified *no partial vendor* decision ("a partial vendor regresses OD capability the moment a later step needs a pruned skill"). The ~80 extra bundles (html-ppt-*, orbit-*, taste variants) are harmless availability, not correctness. Avoids 31 verbose explicit manifest entries + the maintenance burden of tracking which bundles the pipeline references.
- **OQ2 — keep dst or rename.** → **keep `dst: "vendor/open-design/skills/"`.** The dst is an internal vendor convention; keeping it stable means zero edits to the ~7 pipeline references and zero consumer churn beyond the content itself. The truthfulness cost (a path named `skills/` holding `design-templates/` content) is paid by the provenance header (which names the real `design-templates/<bundle>` src) and the manifest src.
- **OQ3 — `kind` label.** → **keep `"skill-bundle-tree"`.** Verified: `kind` is functionally checked only for `'design-system-tree'` (the `isDesignMd` gate, line ~694); `skill-bundle-tree` is display-only (report line ~359). Renaming is cosmetic and risks nothing-gained churn.
- **OQ4 — provenance path.** → **accept the natural change.** `stageFile` builds the header from `srcPath`, so vendored files will read `vendored from open-design@c128ffd5:design-templates/<bundle>/…` — correct; provenance should name the real upstream path. This is also why the 31 bundles' bytes change (header) even where content is identical.

## Alternatives considered

### Vendor only the 31 pipeline bundles (31 explicit recursive manifest entries)
Rejected — smaller blast radius (~150 files vs 729) but contradicts spec 027's ratified no-partial-vendor stance, bloats the manifest with 31 entries, and creates a silent-failure maintenance burden (a future pipeline step references a bundle not in the 31 → breaks). The whole-tree mirror is the established model.

### Rename dst to `vendor/open-design/design-templates/` (truthful path)
Rejected — would force edits to every `vendor/open-design/skills/<bundle>` reference across `02-prototype/{prompt.md, references/od-bridge.md, references/pipeline.md}` AND re-propagate to 4 repos for a cosmetic gain. Provenance + manifest src already carry the truth.

### Fix mapping + prune orphans in one spec
Rejected — keeps 143's diff reviewable (mapping + content) separate from 142's destructive prune logic. The debate established the two as predecessor/successor; merging them re-buries the destructive-deletion design under a large content diff.

## Risks and unknowns

- **Large vendor diff (~729 files) committed.** Real but inherent to the remap; this spec commits it to Agent0 only (consumer re-sync is a sequenced follow-up after 142).
- **`--verify` stays red after this spec** (skills/ orphans from the old creative set). EXPECTED and documented — 142 makes it green. 143's validation is "the 31 bundles resolve from design-templates," not "verify passes."
- **design-systems/ re-staged by the same `--apply`** — should be byte-identical (same sha/provenance) → no git diff there. Confirm at validation.
- **Cached tarball assumption** — if `runtime/od-sync/extracted-c128…` were cleared, `--apply` re-downloads. Acceptable (one-time), but note it.
