Order matters: A unblocks B (sub-agent briefs reference bundled template paths); B unblocks C (template fixes need the new skill structure to anchor against); each pass ends with a `/skill validate prototype` exit-0 gate. After C, run live dogfood (replay the 2026-05-17 "Claude Code governance dashboard" brief) as the acceptance proof of Scenario 8. Spec 034 superseded marker written last, after dogfood passes.

## Alternatives considered

### Alternative 1 — Keep `/prototype` v1 deprecated, ship `/prototype-v2` lado-a-lado

**Rejected because** of user pick + the 2026-05-17 "skill rejeitada" feedback. v1 had a critical render bug (tokens.css import false-positive) that made the whole design system a no-op AND structural gaps (8 of 13 pipeline steps skipped). Keeping v1 around as deprecated would (a) clutter the skills catalog with a known-broken artifact, (b) confuse users who invoke `/prototype` and don't know there's a `/prototype-v2`, (c) double the dogfood surface for both forks of Agent0 to maintain. The in-place refactor with `**Status:** superseded` on spec 034 is cleaner.

### Alternative 2 — Use `mcp__product-pipeline__*` as the backend

**Rejected because** of user pick. The MCP-backed approach (the original recommendation in the proposal) would have inherited schema validation + state machine + design-systems catalog for free, with the trade-off of requiring the MCP server to be running. The user chose standalone (templates copied) to avoid that runtime dependency, accepting the trade-off of quarterly drift sync. This decouples `/prototype`'s portability from MCP server availability — a valid call for the fork-portability story (skills should be cc-native per spec 033's portability tier, not runtime-locked to a co-located MCP).

### Alternative 3 — Three-tier depth flag (`--fast`/`--standard`/`--deep`)

**Rejected because** of user pick. Three tiers × 13 sub-agents = 39 distinct prompt templates to maintain, with thin marginal value over a well-calibrated single tier. The single-tier shape forces the design discipline ("what's the right depth for a startup founder validating an MVP?") instead of deferring it to runtime. Users wanting more rigor invoke the MCP pipeline directly; users wanting less are out of scope (the skill ALREADY skips/lightens vs the MCP heavy mode).

### Alternative 4 — Auto-pass phase gates (no user intervention)

**Rejected because** of user pick (Q3 resolution). Auto-passing the 3 gates would make the skill faster but lose the "fail early" property — a wrong direction in Discovery wastes Identity + Specification + Synthesis work. Structured `AskUserQuestion` at each gate provides the audit trail + intent capture the heavy MCP pipeline gates exist to provide; replicating that discipline in agile mode is what makes v2 a real frontend (not a degenerate one).

### Alternative 5 — Per-step retry loop with brief augmentation (Q4)

**Rejected because** of user pick (Q4 resolution). The retry-with-augmented-brief pattern (cap 3 retries) would be more robust against transient sub-agent failures, but adds significant orchestration logic (failure-mode detection, brief augmentation rules, loop budget tracking) for a low-frequency event. Degrade-gracefully + log + continue is simpler and matches the spec 034 Phase 3 BLOCKED pattern that already works.

## Risks and unknowns

- **Risk 1 — Template bundle size.** Copying 13 step templates verbatim adds ~150-300 KB to `.claude/skills/prototype/` (estimated from `packages/mcp-product-pipeline/src/templates/` directory size). Skills validator (spec 033) has a 5000-token body cap on SKILL.md itself, but no cap on template bundle. Sub-risk: the OD catalog index (Step 6) could push this to ~500 KB if we copy DESIGN.md files too. Mitigation: only copy the index JSON; per-vendor DESIGN.md stays in the package + Step 6 brief falls back to custom path if the package is absent.

- **Risk 2 — Sub-agent dispatch latency for Phase 1.** Step 1 = opus + web research = high token cost + long wall-clock (Explore agent example: ~35K tokens, 90s; opus Step 1 could be 50-80K tokens + 2-3 min). 13 steps total = potentially 30-45 min real wall-time even when Phase 1/2/3/4 each parallelize internally. Mitigation: clear handoff message at each phase gate so user knows what's happening; consider showing per-step `[in_progress]` status messages.

- **Risk 3 — Phase 1 web search depth calibration.** "standard" tier for Step 1 says 5-8 web searches (vs `deep` 15-25). The 5-8 number is unvalidated — could be too few for genuine market discovery on novel ideas, too many for well-trodden product spaces. Mitigation: per Q1, tune via dogfood, not upfront — initial spec value is "5-8 searches" with explicit "tunable" annotation in the Step 1 brief.

- **Risk 4 — Template-drift sync miss.** Quarterly REMINDERS is low-ceremony but high-trust — depends on the human reading REMINDERS at session-start and acting on it. If 2 quarters pass without sync, the bundled templates drift behind the canonical MCP pipeline. Mitigation: REMINDERS due date acts as forcing function; first miss triggers a discussion on whether to escalate to spec 016 sync-harness script (Q2 alternative B).

- **Risk 5 — `/sdd new <slug>` handoff message accuracy.** v2 Phase 4 closes with `/sdd new <slug>` — but the v2 prototype now contains FAR more than v1 (concept-brief, full spec, system-design, etc). The `/sdd new` template assumes the user is starting from scratch and will fill spec.md by hand. We may want to teach `/sdd new` (or a sibling skill) to import these v2 artifacts into the new spec dir. Out of scope for spec 036 but flag as Risk 5.

- **Unknown 1 — Whether the OD vendor catalog actually exists at `packages/mcp-product-pipeline/design-systems/`.** Asserted in plan but not verified. Implementation must check before reading. If absent, Step 6 catalog path is disabled and `delegation-briefs.md` instructs the sub-agent to use custom-tokens path only.

- **Unknown 2 — Whether bundle copy + step-brief inline content fits in the SKILL.md body cap.** Spec 033 enforces ~5000 tokens for SKILL.md body. v1 SKILL.md is 130 lines (~2900 tokens). v2 will likely push past 5000 if we inline per-step brief logic. Mitigation: keep SKILL.md as 4-phase orchestration body; per-step briefs live in `delegation-briefs.md` (not SKILL.md). Validator should still pass — but we'll verify in pass B and refactor inline if needed.
