## Notes

The implementation was dogfooded: tasks 3, 5, 6, 8 were delegated via real `Agent` dispatches with the 5-field discipline, mimicking what the spec enforces post-implementation. The dogfood surfaced two findings worth carrying forward:

1. **Override marker false-positive on prose mentions** — discovered on the very first real dispatch (task 8, the rule writer). The brief contained the literal string `# OVERRIDE: <reason ≥10 chars>` as part of explaining the marker, and the unanchored regex captured it as a real bypass. Fixed in two ways: (a) anchored the marker to start-of-line with optional leading whitespace; (b) restructured so override only suppresses the *block* — the 5-field check always runs and the audit `formatted` field reflects reality. `plan.md § Override marker` updated with the decision.

2. **Two non-obvious bash bugs surfaced during sub-agent implementation of post-edit-validate.sh** (caught and fixed by the implementing sub-agent before deliverable):
   - `jq '.field // empty'` collapses `false` and missing into the same empty string. When checking `ok` from validator output, use `if type=="object" and has("ok") then (.ok|tostring) else "" end`.
   - `exec 9>"$LOCK_PATH" 2>/dev/null` is a sticky redirect — it permanently silences stderr for the rest of the script. Probe writability in a subshell first.

   Both gotchas now live in `.claude/rules/delegation.md` for future maintainers.

The post-edit validator is currently inert in this base repo (no language stack → `no-stack-detected` → ok=true), which is by design: the validator is project-replaceable, and a real project will plug in its own. The audit and advisory pipelines are fully exercised even without the validator doing real work.
