# 073 — product-report-html

_Created 2026-05-21._

**Status:** shipped

## Intent

`/product` produces ~15 markdown artifacts plus visual-contract HTML under `<out>/docs/`. Today the only navigable surface is `REPORT.md` — a markdown narrative a human reads in a text editor, jumping out to each `concept-brief.md` / `functional-spec.md` / etc. as raw markdown. That is poor reading UX, and the pain bites hardest at the three `AskUserQuestion` gates, where a decision depends on having actually read the artifacts. This spec adds `docs/REPORT.html`: a single self-contained, navigable HTML reading surface that renders every pipeline artifact in pipeline order, generated by a deterministic script invoked at four moments (the 3 gates + terminal). `REPORT.md` stays — it is the git-tracked narrative; `REPORT.html` is the read-don't-edit surface built on top of it.

## Acceptance criteria

- [ ] **Scenario: terminal report renders the full pipeline**
  - **Given** a completed `/product` run with all 15 artifacts present under `<out>/docs/`
  - **When** `build-report.ts --out=<out>/docs` runs
  - **Then** `<out>/docs/REPORT.html` exists, opens in a browser with no server/build, and its sidebar lists all 15 steps grouped by the 5 phases in pipeline order

- [ ] **Scenario: partial run at a gate**
  - **Given** a `/product` run paused at gate 1 with only steps 01-04 artifacts present
  - **When** `build-report.ts` runs
  - **Then** `REPORT.html` renders the steps 01-04 artifacts and shows steps 05-15 as a greyed-out "not yet generated" placeholder — no error, no crash

- [ ] **Scenario: regeneration is idempotent**
  - **Given** `REPORT.html` already exists from a prior run
  - **When** `build-report.ts` runs again against the same unchanged `docs/`
  - **Then** the regenerated `REPORT.html` is byte-identical to the previous one, except for the single declared `generated_at` field (no duplication, no other churn)

- [ ] **Scenario: markdown is rendered, not raw**
  - **Given** an artifact such as `concept-brief.md` containing headings, tables, and fenced code
  - **When** that artifact is selected in `REPORT.html`
  - **Then** it displays as rendered HTML — headings, tables, syntax-highlighted code — not raw markdown text

- [ ] **Scenario: visual-contract HTML embedded inline**
  - **Given** `docs/screens/*.html` (lo-fi) and `docs/screens/hifi/*.html` (hi-fi) exist
  - **When** the Step 02 / Step 15 entries are selected
  - **Then** the mood screens render inline via `<iframe>` with a relative `src`

- [ ] **Scenario: blocked step surfaced**
  - **Given** `docs/.state.json` lists a step in `blocked_steps`
  - **When** `REPORT.html` loads
  - **Then** that step's nav entry shows the `✗` blocked status icon, consistent with `REPORT.md` § Pipeline coverage

- [x] **Scenario: deep-link and history navigation** _(QA 073, post-ship)_
  - **Given** `REPORT.html` is open
  - **When** the URL hash changes — via a cold load, an address-bar edit, or the browser back / forward buttons
  - **Then** the artifact named by the hash is rendered (a `hashchange` listener drives navigation; nav clicks set `location.hash`)

- [x] **Scenario: usable on a narrow viewport** _(QA 073, post-ship)_
  - **Given** `REPORT.html` open at a viewport ≤ 720px wide
  - **When** the page loads
  - **Then** the sidebar collapses to an off-canvas drawer toggled by a `☰` button (with a tap-to-close backdrop), the content pane is full-width, and there is no horizontal document overflow

- [x] **Scenario: a multi-part step is navigated by sub-tabs** _(post-ship, 2026-05-22)_
  - **Given** a step with more than one artifact part (e.g. Step 15 — Hi-fi screens, screen-atlas, fixture-spec)
  - **When** that step is opened in `REPORT.html`
  - **Then** a sub-tab row renders at the top of the pane, only the active tab's parts are in the DOM (a long step is never one giant scroll), and each tab is deep-linkable via a `#<step>/<tab-slug>` hash that survives back / forward

- [ ] `.claude/skills/product/scripts/build-report.ts` exists and runs with the same runtime as `sync-open-design.ts`, with **zero npm dependencies** (Node/Bun stdlib only — no markdown parsing at build time)
- [ ] `.claude/skills/product/templates/report.html.tmpl` exists — the HTML shell (CSS + client JS + pinned CDN `<script>` tags + placeholders)
- [ ] `.claude/skills/product/scripts/build-report.test.ts` exists and passes
- [ ] `.claude/skills/product/SKILL.md` invokes `build-report.ts` at all four moments: before gates after steps 04 / 12 / 14, and at the terminal Step 15.4 alongside `REPORT.md`
- [ ] Non-markdown artifacts (`sitemap.yaml`, `data-flow.json`, `tokens.css`) render as syntax-highlighted code blocks

## Non-goals

- **Not a live-reload dev server.** `REPORT.html` is a static file regenerated on demand; `/product` is a batch workflow, so a snapshot regenerated at 4 known moments is sufficient. Live reactivity is deferred until a rule-of-three demand surfaces.
- **Not a markdown editor.** Read-only rendering surface; editing artifacts stays in the editor / pipeline.
- **Does not replace `REPORT.md`.** The git-tracked markdown narrative remains the source of truth and the diffable record; `REPORT.html` is generated on top of it (and renders it as its Overview view).
- **Not offline-first in v1.** Libs load from CDN. Vendoring `marked`/`highlight.js`/`mermaid` into `templates/` for offline use is a documented later option, not v1 scope.
- **Not a generic harness dashboard.** Scoped to `/product` output only — the report knows the fixed 15-step pipeline. No cross-skill artifact viewer.

The three sub-decisions (CDN over vendored libs; artifact manifest as a typed const in the script rather than a separate file; mood screens via relative-path `<iframe>` rather than base64-inline) are recorded in `plan.md` § Alternatives considered and locked._
